## 0. 왜 이 문서가 필요한가

NN/g·Smashing Magazine 정리에 따르면, 빈 상태는 단순 "기록 없음" 표지가 아니라 `상태 설명 → 기대 결과 → 첫 행동` 3박자가 한 화면에서 닫혀야 한다. Virtue는 정식 출시 전이라 실제 전환율 수치는 의사결정급이 아니다. 그래서 빈 상태별로 첫 행동 경로가 닫혀 있는지, `docs/copy-spec.md`의 설교/칭찬 금지선과 충돌이 없는지를 문서로 점검하는 것이 지금 가능한 가장 정직한 작업이다.

이 문서의 산출 단위는 **빈 상태 한 칸 = 한 줄**. 모든 카피 후보는 적용 지시서가 아니라 후보일 뿐이며, 실제 카피 변경은 별도 Intent로 받아 처리한다.

## 1. 현재 빈 상태 인벤토리

| ID | 위치 | 트리거 조건 | 현재 카피 (지금 코드 기준) | 같은 카드 안 CTA | 카드 밖 보조 CTA |
|---|---|---|---|---|---|
| ES-1 | 대시보드 `/` · 덕력 카드 | `stats.count === 0` | 본문 `아직 비어있어요. 오늘 1덕만 시작해볼까요?` | 없음 | 같은 화면 큰 CTA `오늘 덕 쌓기` (Card 외부) |
| ES-2 | 대시보드 `/` · 최근 덕행 섹션 | `recent.length === 0` | `EmptyState` 컴포넌트 / 제목 `아직 기록이 없어요.` / 보조 `오늘 사소한 거 하나, 카메라로 콕.` | 없음 (`action` prop 미사용) | 같은 화면 큰 CTA `오늘 덕 쌓기` |
| ES-3 | 덕행록 `/deeds` | `deeds.length === 0` | `EmptyState` / 제목 `아직 덕이 비어 있네요.` / 보조 `한 컷 찍어두면 여기 차곡차곡 쌓여요.` | 있음 — `action`으로 `Link href="/add"` `덕 쌓기` 버튼 | 하단 nav `덕 쌓기` |
| ES-4 | 덕 쌓기 `/add` · 사진 슬롯 | `!preview` (사진 미선택) | label 내부 / 제목 `오늘의 한 컷, 골라주세요` / 보조 `카메라 또는 갤러리에서` | 있음 — label 전체가 hidden file input의 트리거 | 별도 보조 안내 한 줄 (사진/메모 모두 없을 때) |
| ES-5 | 덕 쌓기 `/add` · 미시도 안내 | `!result && !preview` | `사진은 이 기기에만 잠시 머물러요. 키가 설정돼 있으면 AI가, 없으면 mock이 채점해요.` | n/a (안내 한 줄, 액션 아님) | 같은 화면 `AI 채점`/`임시 판정` 버튼 |
| ES-6 | 환생도감 `/dex` · 잠긴 단계 | `total < s.min` (해당 종 락) | 이모지 `❓`, 이름 `???`, 본문 `다음 생의 비밀.` 또는 `s.nextHint` | 없음 (탐색 전용) | 화면 상단 현재 단계 카드 (행동 아님) |

> 주: ES-1과 ES-2는 **같은 화면에 동시 존재**한다. 첫 방문 직후의 `/`는 ES-1·ES-2가 카드 두 장으로 동시에 보이고, 그 사이에 큰 `오늘 덕 쌓기` CTA가 끼어 있다. 즉 카드 내부 CTA 부재가 곧 첫 행동 경로 단절을 의미하지는 않는다. 단, 두 카드를 동시에 보여줄 때 카피 톤이 비슷하면 같은 말을 두 번 듣는 인상이 될 수 있다 (R1, §3).

## 2. 빈 상태별 감사

각 항목은 `상태 설명 → 기대 결과 → 첫 행동 CTA → 톤 위험 → 계측 후보` 다섯 칸으로 잡는다. 카피 후보는 모두 `copy-spec.md`의 금지어 명단(훌륭한·본받을·모범적인·귀감·인성·미덕·선행·베풂·봉사정신·마음이 따뜻한·좋은 사람·멋진 인격)을 한 글자도 사용하지 않는다.

### 2.1 ES-1 — 대시보드 / 덕력 카드 (count === 0)

| 칸 | 내용 |
|---|---|
| 상태 설명 | 사용자가 아직 덕행을 한 번도 저장하지 않아 `덕력` 합계가 시작값(`INITIAL_VIRTUE`)에 머문다. 카드 본문은 "비어있음" + 첫 행동 권유 한 줄로 닫혀 있다. |
| 기대 결과 | 사용자는 "이 숫자가 비어 있다 = 내가 첫 덕행을 쌓아야 채워진다"고 1초 안에 이해한다. |
| 첫 행동 CTA | 같은 화면 `오늘 덕 쌓기` 버튼이 ES-1과 ES-2 사이에 위치하므로 카드 내부에 별도 CTA를 두지 않는다. 카드 내부 CTA를 추가하면 같은 화면 위/아래에 같은 CTA가 두 번 노출되어 시각 혼잡 + 의미 중복이 발생한다 (Smashing Magazine: empty state는 시각적으로 단순해야 한다). 유지를 권장. |
| 톤 위험 | 현재 본문 `아직 비어있어요. 오늘 1덕만 시작해볼까요?` — 설교/칭찬 0건. "비어있어요"가 자기관찰 톤과 맞고, "1덕만"이 마찰 낮춘다. 다만 ES-2와 두 칸 연속으로 "비어있어요/기록이 없어요"가 반복되어 동어반복 위험이 있다. |
| 계측 후보 (배포 전 보류) | `empty_state_seen` (location=`dashboard_total`)을 추후 정의 가능. 단 본 Intent에서 트래킹 코드 변경 없음. |

### 2.2 ES-2 — 대시보드 / 최근 덕행 섹션 (recent.length === 0)

| 칸 | 내용 |
|---|---|
| 상태 설명 | `최근 덕행` 카드가 비어 있고, 같은 화면 상단에 ES-1, 중간에 큰 `오늘 덕 쌓기` 버튼이 이미 보인다. |
| 기대 결과 | 사용자는 "여기에 내 기록이 시간순으로 쌓일 것"이라는 멘탈 모델을 가진다. NN/g가 말하는 `시스템 상태 + 핵심 작업으로의 경로` 둘 중 **시스템 상태 설명** 비중이 더 크다 (경로는 ES-1 위/아래의 큰 CTA가 이미 담당). |
| 첫 행동 CTA | 두 안:<br>**(A) 유지 — 카드 내부 CTA 없음.** 큰 CTA와 역할 분리. 화면 밀도와 시각 단순성을 우선.<br>**(B) 카드 내부 보조 CTA 추가.** `EmptyState`의 `action` 슬롯에 `Link href="/add"` 텍스트 버튼을 둔다. 단 큰 CTA와 라벨이 같으면 안 됨 (중복). 라벨 후보는 §3에서 분리해 다룬다.<br>권장: 정식 출시 후 `add_flow_started`가 큰 CTA에서 발화하는지 데이터로 보고 다시 결정. 지금은 (A) 유지가 더 정직하다. |
| 톤 위험 | 현재 `아직 기록이 없어요. / 오늘 사소한 거 하나, 카메라로 콕.` — 설교/칭찬 0건. `장난기 + 따뜻` 톤과 맞음. ES-1과의 동어반복은 카피 후보 §3에서 한 단어만 살짝 비틀어 해소 가능. |
| 계측 후보 (보류) | `empty_state_seen` (location=`dashboard_recent`). 또는 ES-2의 보조 CTA 추가 시 `dashboard_recent_cta_clicked`로 큰 CTA와 분리 측정. 본 Intent에서 적용 X. |

### 2.3 ES-3 — 덕행록 / 전체 빈 (deeds.length === 0)

| 칸 | 내용 |
|---|---|
| 상태 설명 | `/deeds`에 직접 진입했지만 누적 기록이 0건이다. 큰 화면을 전부 차지하는 빈 상태 카드만 보이고 그 위는 작은 헤더 한 줄뿐이다. |
| 기대 결과 | 사용자는 "여기는 누적 보관함이고, 지금 비어 있으니 곧장 덕 쌓기 화면으로 가야 한다"고 인식한다. 대시보드와 달리 같은 화면에 큰 CTA가 따로 없다. |
| 첫 행동 CTA | **유지**: `EmptyState`의 `action`으로 `덕 쌓기` 버튼이 이미 존재. 단독 화면 빈 상태에서 카드 내부 CTA는 필수에 가깝다 (NN/g: "직접 경로 제공"). 라벨도 하단 nav와 동일한 `덕 쌓기`로 일치되어 인지 비용이 낮다. 변경 권장 사항 없음. |
| 톤 위험 | `아직 덕이 비어 있네요. / 한 컷 찍어두면 여기 차곡차곡 쌓여요.` — 설교/칭찬 0건, "차곡차곡" 의성/의태가 따뜻 톤과 맞음. 보조 카피의 "여기 차곡차곡 쌓여요"가 시스템 상태 + 기대 결과를 한 번에 전달. |
| 계측 후보 (보류) | `empty_state_seen` (location=`deeds_list`), `deeds_list_cta_clicked` (CTA 클릭 후 `/add`로 이동) — 필요 시 추후 정의. 현재 트래킹은 `add_flow_started`/`deed_judge_attempted`/`deed_saved`까지 이미 존재. |

### 2.4 ES-4 — 덕 쌓기 / 사진 슬롯 (!preview)

| 칸 | 내용 |
|---|---|
| 상태 설명 | `/add` 진입 직후 사진 슬롯이 비어 있고 메모 입력란이 빈 상태다. 카드 전체가 hidden file input의 라벨이므로 슬롯의 어디를 눌러도 카메라/갤러리가 열린다. |
| 기대 결과 | 사용자는 "사진 하나 고르면 그 다음 단계로 흐른다"는 멘탈 모델을 가진다. 즉 이 빈 상태는 거의 그 자체가 첫 행동 트리거다. |
| 첫 행동 CTA | **유지**: 카드 자체가 클릭 가능 영역. `오늘의 한 컷, 골라주세요`가 행동 안내. 보조 `카메라 또는 갤러리에서`가 어떤 트리거가 열리는지 미리 알려준다. 사진 없이도 메모만으로 채점 시도가 가능하지만(에러 카피 후보는 copy-spec에 있음) 이 빈 상태의 1차 권유는 사진 쪽. |
| 톤 위험 | 설교/칭찬 0건. "인증 사진" → "오늘의 한 컷"으로 무게 낮춰진 상태가 이미 적용. 보조 한 줄이 추가로 필요하지 않음. |
| 계측 후보 (보류) | 이미 `deed_judge_attempted`/`deed_judged`/`deed_saved`가 후속 단계를 추적 중. 진입 시점 자체에 대해 `add_view_opened` 또는 `empty_state_seen` (location=`add_photo_slot`) 추가 가능. 본 Intent에서 적용 X. |

### 2.5 ES-5 — 덕 쌓기 / 미시도 안내 (!result && !preview)

| 칸 | 내용 |
|---|---|
| 상태 설명 | 사진도 없고 채점 결과도 없을 때만 보이는 한 줄 안내. 사진을 한 번이라도 고르면 이 카피는 사라진다 (mock/AI 라벨이 결과 카드로 대체). |
| 기대 결과 | 사용자가 "사진은 이 기기 안에서만 쓰임 + AI vs mock 둘 중 하나" 라는 시스템 약속을 1초 안에 안다. |
| 첫 행동 CTA | 안내 한 줄 자체는 CTA 아님. 첫 행동은 사진 슬롯(ES-4) 또는 메모 입력에서 일어남. 변경 권장 사항 없음. |
| 톤 위험 | 설교/칭찬 0건. "키가 설정돼 있으면 AI가, 없으면 mock이"는 톤이 살짝 기술적이지만 prelaunch에서는 정직성이 우선이라 그대로 둔다. |
| 계측 후보 (보류) | n/a (안내 텍스트). |

### 2.6 ES-6 — 환생도감 / 잠긴 단계 (per species: total < s.min)

| 칸 | 내용 |
|---|---|
| 상태 설명 | 도감 카드 중 잠긴 항목. 이모지 `❓`, 이름 `???`, 설명 `다음 생의 비밀.` 또는 `s.nextHint`. 잠금이 풀리려면 누적 덕력이 `s.min` 이상이어야 한다. |
| 기대 결과 | 사용자는 "이 화면은 행동 화면이 아니라 보상 미리보기"라는 사실을 안다. 잠긴 종이 있다는 사실 자체가 회유 장치다. 즉 빈 상태이지만 "이 칸을 채우려면 덕을 쌓아야 한다"는 동기를 짧게 전달하면 된다. |
| 첫 행동 CTA | 카드 내부 CTA 두지 않음. 도감은 탐색/보상 채널이고 같은 화면에 `덕 쌓기` 카드를 두면 도감의 호기심 톤이 깨진다. 첫 행동은 하단 nav `덕 쌓기` 또는 `/`의 큰 CTA가 담당. 유지 권장. |
| 톤 위험 | `다음 생의 비밀.`은 종교적 메타포 안전 범위 내. `❓` + `???`은 미스터리 톤. 잠금 본문에 `훌륭한/모범적인/귀감` 등 도덕 어휘가 새로 들어가면 즉시 위반 — 신규 species 카피 추가 시 검수 필요 (§4-T2 참고). |
| 계측 후보 (보류) | 도감 탐색 행동을 잡으려면 `dex_view_opened`, `dex_locked_card_viewed` 정도가 후보. 본 Intent에서 적용 X. |

## 3. ES-2 카드 내부 CTA 후보 (참고용, 적용은 별도 Intent)

source 노트가 "최근 덕행 빈 상태에 직접 `덕 쌓기` CTA를 둘지" 두 안을 만들어 두라고 했다. 결정은 보류, 후보만 정리한다.

### 후보 A — 카드 내부 CTA **없음** (현재 유지)

- 본문: `아직 기록이 없어요.`
- 보조: `오늘 사소한 거 하나, 카메라로 콕.`
- 액션: 없음. 위/아래에 이미 노출된 큰 `오늘 덕 쌓기` 버튼이 첫 행동을 담당.
- 장점: 화면 밀도 단순. 같은 CTA 라벨이 화면에 두 번 안 보임. Smashing Magazine "empty state는 시각적으로 단순해야 한다" 원칙과 정렬.
- 단점: 같은 카드 안에서 `상태 설명 → 기대 결과 → 첫 행동`이 닫히지 않음. 같은 화면 큰 CTA에 의존.

### 후보 B — 카드 내부 보조 CTA **추가**

- 본문: `아직 기록이 없어요.`
- 보조: `오늘 사소한 거 하나, 카메라로 콕.`
- 액션 후보 라벨: 큰 CTA `오늘 덕 쌓기`와 라벨이 같으면 안 되므로, 다음 중 하나로 분리한다.
  - `여기에 한 컷 넣기`
  - `한 컷 골라보기`
  - `사진 하나 골라보기`
- 장점: 카드 한 장 안에서 시스템 상태 + 첫 행동까지 한 번에 닫힌다. 큰 CTA가 시야에서 벗어나는 작은 화면에서도 첫 행동 경로 손실 없음.
- 단점: 동일 페이지에 두 개의 CTA가 동시에 노출 — 사용자가 잠깐 헷갈릴 수 있음. `EmptyState`의 `action` 슬롯 디자인이 카드 안에서 충분히 가볍게 보이는지 디자이너 확인 필요.

### 결정 가이드 (적용 Intent로 넘길 때)

- prelaunch에서는 후보 A 유지가 안전 (정직성). 큰 CTA가 충분히 잘 보임.
- 실 사용자 데이터에서 `dashboard_recent`까지 스크롤한 후 `add_flow_started` 시작 비율이 유의미하게 떨어지는 패턴이 보이면 후보 B로 전환 후 A/B 비교.

## 5. 계측 후보 정리 (본 Intent 비적용)

source 노트의 후속 후보에 `empty_state_seen`이 명시되어 있다. 본 Intent는 트래킹 변경 권한 밖이므로 **정의만 후보로 정리**하고 코드/PostHog에는 손대지 않는다.

| 이벤트 후보 | 발화 위치 | 핵심 속성 후보 | 정당화 |
|---|---|---|---|
| `empty_state_seen` | ES-1 ~ ES-4, ES-6 (ES-5는 안내 텍스트라 제외) | `location` (`dashboard_total`/`dashboard_recent`/`deeds_list`/`add_photo_slot`/`dex_locked`), `is_first_visit` 추정 플래그 | NN/g 정의대로 빈 상태가 첫 행동 경로의 시작 지점이라는 가설을 확인하려면 노출 자체가 일단 잡혀야 한다. |
| `empty_state_cta_clicked` | ES-2 후보 B 적용 시, ES-3 | `location`, `cta_label` | 카드 내부 CTA와 카드 외부 큰 CTA 중 어느 쪽이 첫 행동을 만들었는지 분리 측정. 후보 B 도입 결정의 근거. |
| `dex_locked_card_viewed` | ES-6 | `stage`, `species_name`, `current_total` | 도감의 빈 칸이 회유 장치로 작동하는지 정도 보기. 우선순위 낮음. |

> 적용 조건: PostHog 이벤트 확장은 별도 Intent로, 프라이버시 영향 점검과 PostHog 대시보드 변경 승인 후 진행.

## 6. 검증 게이트

1. **카피-스펙 금지어 0건**: 본 문서의 모든 카피 후보(§2 현재 카피 인용 + §3 후보 + §4 회피 가드)는 `docs/copy-spec.md`의 금지어 명단(훌륭한·본받을·모범적인·귀감·인성·미덕·선행·베풂·봉사정신·마음이 따뜻한·좋은 사람·멋진 인격)을 한 글자도 사용하지 않는다.
2. **코드 변경 0건**: 본 Intent로 `src/`, `public/`, `next.config.*`, `.env*`, PostHog 키/대시보드 어떤 것도 변경하지 않는다.
3. **첫 검증 grep**: Intent가 요구한 다음 명령으로 근거 인용과 충돌 가능 어휘를 확인한다.
   ```bash
   rg -n "설교|칭찬|empty_state_seen|덕 쌓기|최근 덕행|덕행록" \
     docs/empty-state-first-action-audit.md docs/copy-spec.md src/app src/components
   ```
   - 이 문서에 "설교", "칭찬"이 등장하는 경우는 모두 **금지선을 설명/회피하기 위한 메타 언급** 한정이며, 카피 후보 본문에는 등장하지 않는다.
   - `empty_state_seen`, `덕 쌓기`, `최근 덕행`, `덕행록`은 본 문서·`copy-spec.md`·코드 양쪽에서 합법적으로 사용되는 어휘이며, 충돌이 아니라 정합성 확인 신호다.

## 7. 비수행 명시

- 외부 발송 0건 (메일·푸시·SMS·SNS·웹훅·DM 어느 채널도).
- 배포 0건 (`https://virtue.oracle.shdkej.com` 또는 다른 환경에 새 빌드 없음).

- 시크릿/권한/IAM 변경 0건.
- `src/`, `api/` 등 런타임 코드 변경 0건. `docs/` 한 파일만 추가.
- 다른 사람에게 새 메시지/리뷰/이슈 작성 0건.

## 8. Out of scope (별도 Intent로 분리해야 할 후속)

- ES-2 카드 내부 CTA 추가 결정 (후보 A vs B 중 택1) — Marketer + Planner 합의 + 실제 빌드 적용 Intent로 분리.
- `empty_state_seen` 외 신규 PostHog 이벤트 도입 — 트래킹 변경 승인 후 별도 Intent.
- `/me`에서 `clearDeeds()` 직후의 빈 상태 카피 추가 — 별도 회수 시나리오. 본 감사 범위 밖.
- 데일리 캡(`30덕 상한`) 도달 시 카피는 "빈 상태"가 아니라 "상태 한계"이므로 본 문서 범위 밖.
