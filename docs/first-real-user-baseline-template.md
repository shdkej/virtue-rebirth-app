# Virtue 첫 실사용자 기준선 기록 템플릿

## 0. 목적 + 전제

### 목적

이 문서는 Virtue가 만나는 **첫 실사용자 10~20명을 한 명씩 같은 칸에 받아 적기 위한 빈 기록 양식(fill-in form)**이다. 선행 두 문서가 *개념과 계산*을 정의했다면, 이 문서는 그 정의를 실제 사람에게 적용해 *한 줄씩 채워 넣는 운영 표*다. J1~J4 어떤 잡을 만나든, 누가 기록하든, 출시 학습이 같은 형식으로 쌓이도록 칸을 고정한다.

세 문서의 역할을 섞지 않는다.

- `docs/activation-milestone-ladder.md` = **개념 매트릭스.** J1~J4 × setup/aha/habit 칸의 정의서.
- `docs/time-to-value-observation-brief.md` = **시간 간격 계산 기준표.** first/second value까지의 time gap 계산 방식.
- 본 문서(marketing-11) = **빈 기입 양식.** 위 두 문서의 정의를 *재정의하지 않고 계승*해, 실제 사용자 한 명을 한 행으로 받아 적는다. 정의를 바꾸는 부분이 있다면 그것은 충돌이며, 본 문서는 *칸과 작성법*만 새로 한다.

### 전제 못박기

1. **prelaunch이므로 이 표는 성패 채점표가 아니라 관찰 기록부다.** 한 행이 채워졌다고 활성화 성공이 아니고, 비어 있다고 실패가 아니다. "누가 어떤 약속을 보고 들어와 어디까지 갔는가"를 *적을* 뿐, 빠르다/느리다, 좋다/나쁘다를 선언하지 않는다.
2. **새 트래킹 / 이벤트 / 속성 / 코드 / 대시보드 / 비용은 0이다.** 기존 발화 이벤트 `add_flow_started`, `deed_judged`, `deed_saved`, `level_up_viewed`만 사용한다. 새 측정 수단을 만들지 않고, 표의 모든 값은 기존 이벤트와 정성 메모로만 채운다.
3. **외부 발송 / 공개 모집 / 인터뷰 요청은 하지 않는다.** 유입 문장과 정성 메모는 이미 허용된 대화나 내부 관찰에서 확인한 내용을 적는 칸이지, 본 Intent로 보내는 메시지가 아니다.

## 1. 왜 외부 평균이 아니라 자기 기준선인가

Virtue는 아직 정식 오픈 전이다. 이 단계에서 가장 흔한 실수는 외부에서 검증된 온보딩 평균을 그대로 빌려와 성패 기준으로 삼는 것이다. Userpilot 2025 벤치마크의 SaaS activation 평균 약 37.5%, time-to-value 평균 약 1일 12시간 같은 수치는 잘 정리된 참고점이지만, 표본이 B2B SaaS 중심이고 Virtue의 잡(J1~J4)이나 prelaunch 맥락과는 다른 세계에서 나온 숫자다. 첫 10~20명이라는 작은 표본에 테스트 트래픽이 섞이기 쉬운 우리 상황에서, 이런 외부 평균을 목표로 걸면 잘못된 압박만 생긴다. "37.5%에 못 미쳤으니 온보딩이 실패했다" 같은 결론은 표본이 그 판정을 견디지 못하므로 그 자체로 오독이다.

그래서 이 문서는 다른 출발점을 택한다. 외부 평균을 *목표*로 삼는 대신, 우리 첫 실제 사용자가 실제로 어떻게 들어와 어디서 가치를 느꼈는지를 먼저 *기록*한다. 이 기록이 곧 Virtue의 자기 기준선이다. 산업 평균보다 정직한 비교 대상은 언제나 자기 제품의 직전 모습이며, 기준선은 출시 후 개선이 같은 방향으로 움직이는지를 보기 위한 출발점이지 출시 전 성적표가 아니다. 외부 수치는 "이런 방향이 있다"는 방향 참고용으로만 쓰고, prelaunch 성패 판정에는 절대 끌어오지 않는다. 첫 10~20명을 전환율·리텐션·PMF·세그먼트 크기로 판정하지 않으며, 40%나 외부 임계값을 적용하지 않고, 한 명의 강한 표현을 제품 방향으로 확정하지 않는다.

**컬럼 작성법 (한 줄 가이드)**

- **관찰 대상**: 내부 식별자(U-001 등)만. 실명·연락처·외부 공유 금지.
- **유입 문장**: 사용자가 *어떤 약속/말*을 보고 들어왔는가. 작성법은 §3을 따른다. 모르면 "(미확인)"으로 둔다.
- **추정 잡**: 첫 세션의 주된 동기 하나. 헷갈리면 "혼합/미정"으로 둔다(억지 분류 금지).
- **first value / second value**: 잡별 매핑은 §2-1을 따른다. 이벤트명 + 시각을 그대로 적고, 없으면 "미관찰". 시각 차이(time gap) 계산이 필요하면 `time-to-value-observation-brief.md` §2를 따른다 — 이 표는 시각만 받아 적는다.
- **synthetic/test 여부**: QA·내부 점검·자동 입력 등 실사용자가 아닌 트래픽은 **test (제외)**로 표시한다. 판정 기준은 §4. 제외 행은 기준선 해석·표본에서 빼되, 지우지 말고 남겨 둔다(왜 제외했는지 추적 가능하도록).
- **판정 보류 메모**: 성패/리텐션/PMF 결론을 쓰지 않는다. *해석하면 안 되는 점*만 남긴다.

### 2-1. J1~J4 분류 + first/second value 매핑

전체 정의는 `docs/first-session-jtbd-matrix.md`와 `docs/activation-milestone-ladder.md`에 위임한다. 여기서는 *어느 칸에 적을지 빠르게 고르기 위한 한 줄 기준*과, 각 잡의 가치가 "무엇을 느끼면 닫히는가"를 이벤트와 함께 둔다(선행 문서와 동일한 매핑).

| 잡 | 한 줄 분류 기준 | first value — 무엇을 느끼면 닫히나 (이벤트) | second value — 무엇을 한 번 더 느끼면 닫히나 (이벤트) |
|---|---|---|---|
| **J1 기록형** | "오늘 한 거 즉시 남기러" — 저장 자체가 목적 | "방금 한 일을 남겼다"는 즉시 만족 / 첫 `deed_saved` | "또 남길 수 있다"가 확인됨 / 다음 `deed_saved` |
| **J2 누적형** | "얼마나 쌓였나" — 점수·환생종 누적감이 목적 | "쌓였다"는 누적감 / 첫 `deed_saved` | 누적이 한 번 더 보임 / 다음 `deed_saved`, 보조로 `level_up_viewed` |
| **J3 AI 호기심형** | "AI가 어떻게 보나" — 채점 확인이 목적, 저장은 선택 | "AI가 내 행동을 이렇게 봤구나" / 첫 `deed_judged` (저장 전이라도 닫힘) | "다르게 보면?" 재확인 또는 저장으로 이어짐 / 다음 `deed_judged` 또는 `deed_saved` |
| **J4 회고형** | "나중에 돌아볼 로그 만들러" — 지연 가치가 목적 | "나중에 돌아볼 게 생겼다"는 지연 가치 / 첫 `deed_saved` | "쌓이면 로그가 된다"가 확인됨 / 다음 `deed_saved` |

> J1·J2·J4의 first value는 모두 `deed_saved`지만 가치 방향이 다르다(J1 즉시 만족 / J2 누적감 / J4 지연 가치). J3만 `deed_judged`에서 first value가 닫히며, 저장 없이 종료해도 이탈이 아니다. J2의 `level_up_viewed`는 보조 신호일 뿐, 첫 세션에 도달하지 않았다고 누적 가치가 없다고 단정하지 않는다.

## 3. 유입 문장(acquisition/referral phrase) 기록 가이드

기준선의 앞 칸은 "누가 어떤 약속이나 말을 보고 들어왔는가"다. 이 한 칸을 따로 남기는 이유는, 같은 J1 기록형이라도 "사소한 거 하나 남길 수 있다더라"를 듣고 온 사람과 "AI가 내 행동을 점수 매겨준대"를 보고 온 사람은 첫 가치에 닿는 경로가 다를 수 있기 때문이다. 유입 문장은 우리가 어떤 약속을 했을 때 누가 반응했는지를 보여주는, 가장 앞단의 정성 신호다. 어떤 행동 이벤트보다 먼저 발생하며, 나중에 first value 도달과 겹쳐 읽으면 "이 약속을 보고 온 사람은 무엇을 가치로 느꼈나"를 한 줄로 잇게 해준다.

기록 방식은 단순하다. **사용자 본인이 쓴 표현을 그대로 옮긴다.** "친구가 매일 하나씩 적는 거 보여줘서"처럼 들은 그대로 적고, 우리 말로 다듬거나 마케팅 카피처럼 정돈하지 않는다. 본인 표현을 직접 확인하지 못해 우리가 짐작해 채운 칸은 반드시 끝에 "(추정)"을 붙여, 관찰값과 추측을 한눈에 구분되게 한다. 유입 경로를 전혀 모르면 비워두고 "미확인"으로 남기지, 그럴듯한 문장을 지어 넣지 않는다.

채우는 출처도 못박는다. 이 칸은 **이미 허용된 대화나 내부 메모에서만** 채운다. 본 문서를 근거로 외부에 메일·DM·설문·푸시를 보내 유입 경로를 묻는 일은 하지 않는다. 즉 "유입 문장 수집"이 새로운 외부 접촉을 정당화하지 않는다 — 자연스럽게 이미 들어와 있는 말만 받아 적는다.

이 표는 사용자에게 노출할 카피를 만드는 자리가 아니다. 유입 문장 칸도 사용자의 말을 *받아 적는* 자리이지 우리가 *내보낼* 문구를 짓는 자리가 아니다. `docs/copy-spec.md`의 금지어를 사용하지 않으며, 화면·메시지 카피를 새로 정의하지 않는다.

## 4. synthetic / test traffic 제외 기준

이 기준선은 "실제로 무언가를 남기러 온 사람"만 해석에 넣는다. prelaunch에서는 우리 손으로 만든 트래픽이 표본의 대부분일 수 있으므로, 운영 관점에서 먼저 할 일은 **실제 사용자와 테스트/합성 트래픽을 사람 판단으로 분리**하는 것이다. 새 이벤트나 속성을 만들지 않고, 기존 4개 이벤트(`add_flow_started` / `deed_judged` / `deed_saved` / `level_up_viewed`)의 발화 패턴과 우리가 아는 맥락만으로 구분한다.

제외 신호는 다음과 같이 본다. 어느 하나라도 명확하면 그 행을 "test (제외)"로 표시한다.

| 제외 신호 | 어떻게 알아채나 (기존 이벤트 + 사람 판단) |
|---|---|
| 본인·팀 내부 계정 | 우리가 아는 내부 식별자/계정. 데모·시연·스크린샷용 진입 포함. |
| QA·디버그 세션 | 기능 점검 목적으로 들어간 세션. 같은 화면을 의도적으로 왕복한 흔적, 디버깅 중 발생한 진입. |
| 동일 디바이스 반복 테스트 | 같은 환경에서 짧은 간격으로 `add_flow_started`가 여러 번 반복되며, 실제 행동 기록이 아니라 동작 확인으로 보이는 경우. |
| 명백한 더미 입력 | "test", "asdf", "ㅁㄴㅇㄹ", 의미 없는 한 글자 등 행동 기록으로 볼 수 없는 입력. |
| 짧은 시간 다중 `add_flow_started` 폭주 | 한 사람이 매우 짧은 시간 안에 입력 플로우만 연속 진입한 흔적. 가치 경험이 아니라 부하/동작 확인일 가능성. |
| 봇·크롤러 | 사람의 입력 흐름으로 보기 어려운 자동화된 진입. |

운영 원칙은 **삭제가 아니라 표시**다. 기준선 표의 `synthetic/test 여부` 컬럼에 제외 대상을 표시하되, 행 자체는 그대로 남기고 비율·평균 같은 집계에서만 빼고 읽는다. 제외 사유도 정성 메모에 한 줄로 같이 적는다(예: "내부 QA", "더미 입력"). 이렇게 두는 이유는 두 가지다 — (1) 나중에 "왜 이 행을 뺐는지"를 다시 추적할 수 있어야 하고, (2) 제외 판단이 잘못됐을 때 되돌릴 수 있어야 한다. 애매하면 제외하지 않고 "판단 보류"로 표시한 뒤, 주간 리뷰(§6)에서 한 번 더 본다.

이 분리는 자동 규칙이 아니라 사람이 내리는 판단이다. prelaunch 단계에서 표본이 작을수록 합성 트래픽 한두 건이 비율을 통째로 흔들기 때문에, 숫자를 보기 전에 이 컬럼을 먼저 채우는 것이 순서다.

## 5. 최소 표본 조건

이 절은 **합격선이 아니라, 수치 해석을 시작해도 되는지를 가늠하는 prelaunch 관찰 기준**이다. 아래 조건을 못 채웠다고 실패가 아니라, "아직 비율·평균으로 말하지 않는다"는 뜻일 뿐이다.

- **제외 후 실제 사용자가 충분히 모이기 전에는 비율·평균을 말하지 않는다.** 첫 10~20명 단계, 특히 제외 컬럼을 걷어낸 실제 사용자가 한 자릿수일 때는 "전환율 N%", "평균 first value time" 같은 표현을 쓰지 않는다. 대신 "누가 어떤 잡에서 어디까지 갔다"를 사람 단위로 읽는다(`activation-milestone-ladder` §4 작은 표본 금지선 계승).
- **잡 분포가 한쪽으로 쏠리면 잡별 결론을 보류한다.** 실제 사용자가 J1~J4 중 한두 잡에만 몰려 있으면, 비어 있는 잡에 대해서는 first/second value 경향을 말하지 않고 "표본 없음"으로 둔다. 특히 J3(AI 호기심형)은 `deed_saved` 없이 정상 종료할 수 있으므로, J3 표본이 적을 때 저장 부재를 이탈로 합산하지 않는다(`time-to-value` §1, ladder §2 계승).
- **한 사람의 강한 신호를 제품 방향으로 확정하지 않는다.** 표본이 작을수록 한 명의 인상적인 경험이나 막힘이 과대평가되기 쉽다. 기준선에는 기록하되, 방향 결정의 근거로 굳히지 않는다.
- **median/평균을 단독으로 신뢰하지 않는다.** 표본이 작으면 중앙값도 흔들린다. 수치보다 "오래 걸린 세션이 어디서 멈췄나"라는 정성 관찰을 우선한다(P90/꼬리 관찰은 §6과 TTV 문서가 다룬다).

요약하면, **제외 후 실제 사용자 수가 작거나 잡 분포가 쏠려 있으면 — 비율·평균은 보류하고 사람·상황 단위 관찰만 남긴다.**

## 6. post-launch 주간 리뷰 리듬 초안

출시 후에는 이 기준선을 **가벼운 주간 리듬**으로 재검토한다. 무거운 분석 체계가 아니라, 표를 살아 있게 유지하는 운영 루틴이다.

- **누가 채우나** — 운영자가 한 주 동안 들어온 실제 사용자 행을 기준선 표에 추가하고, synthetic/test 제외 플래그를 먼저 채운다. 잡 추정과 정성 메모는 기획·마케팅 관점과 함께 보완한다.
- **언제 모아 보나** — 주 1회, 짧게. 그 주에 새로 들어온 행만 훑고, 이전 주에 "판단 보류"로 둔 행을 다시 본다.
- **synthetic 제외 재확인** — 매주 제외 컬럼을 다시 점검한다. 지난주에 실제 사용자로 분류했지만 더미로 드러난 행, 반대로 과하게 제외한 행을 바로잡는다. 제외 기준 자체에 새로 추가할 신호가 생겼는지도 함께 본다.
- **median뿐 아니라 P90·오래 걸린 세션 정성 점검** — 가운데 값만 보지 않고, 유난히 오래 걸린 세션을 골라 "어디서 막혔나"를 정성으로 본다(꼬리 관찰). 수치 자체보다 막힘 지점의 패턴을 모으는 것이 목적이다.

다만 이 리듬은 **문서상 초안**이다. 실제 대시보드 생성, 자동 집계, 정기 알림 발송 같은 자동화는 모두 **Waiting / approval-needed**이며, 본 문서로 만들지 않는다(ladder §5 후속 경계 계승). 출시 전까지는 수기 관찰표 한 장으로 충분하다.
