# Virtue 첫 주 활성화-리텐션 연결표

## §0 목적 + 전제

### 목적

첫 주 활성화(first value)를 **7일 내 재가치(second value)로 연결**해, 리텐션의 *선행 신호*를 관찰한다. 선행 3문서가 각각 사다리(개념)·time gap(계산)·기입 양식(행)을 정의했고, 본 문서는 그것을 **"first value → 7일 내 second value" 한 줄 연결**로 재합성한다. 정의를 바꾸면 충돌이며, 본 문서는 **연결 관점만** 새로 더한다.

### 세 문서 역할 분리

| 문서 | 역할 | 본 문서와의 관계 |
|---|---|---|
| `activation-milestone-ladder.md` (m09) | setup→aha→habit 사다리 × J1~J4 **개념 매트릭스** | 개념 원천. 그대로 계승 |
| `time-to-value-observation-brief.md` (m10) | first/second value **time gap 계산 기준표** | 계산은 전적으로 이 문서에 위임 |
| `first-real-user-baseline-template.md` (m11) | 첫 10~20명 **한 명당 한 행 기입 양식** | 행 양식 그대로 계승 |
| **본 문서 (m14)** | first value → 7일 내 second value **연결 렌즈** | 위 셋을 잇는 한 줄 연결만 새로 함 |

### 전제 (못박기)

1. **관찰 기록부**: prelaunch이므로 성패 채점표가 아니다. 첫 주 행동을 관찰·기록하는 기록부일 뿐, 어떤 값도 "좋다/나쁘다"로 판정하지 않는다.
2. **신규 0**: 신규 트래킹·이벤트·속성·코드·대시보드·비용을 단 하나도 만들지 않는다. 기존 4개 이벤트(`add_flow_started`, `deed_judged`, `deed_saved`, `level_up_viewed`)만 사용한다.
3. **외부 액션 0**: 외부 발송·공개 모집·인터뷰 요청을 0건 수행한다. 본 문서는 내부 관찰 렌즈에 한정된다.

## §1 왜 획득이 아니라 첫 주 가치 연결인가

신규 사용자를 더 데려온다고 리텐션이 따라오지는 않는다. Amplitude의 B2B 제품 벤치마크와 ProductLed의 PLG 벤치마크가 공통으로 말하는 핵심은, 첫 주의 리텐션 곡선은 획득량이 아니라 **사용자가 초기에 핵심 가치 순간(activation point)에 얼마나 일찍 닿았는가**에서 갈린다는 것이다. activation을 일찍 관찰하지 못한 채 유입만 늘리면, 곡선은 가파르게 떨어진 뒤 평평해지지 않는다. 벤치마크 수치 자체(전환율·activation rate 평균선)는 B2B SaaS 표본에서 나온 방향 참고점일 뿐, Virtue prelaunch 첫 10~20명의 합격선이 아니다 — 이 숫자를 목표로 걸지 않는다.

그래서 지금 Virtue에서 의미 있는 마케팅 일은 유입을 늘리는 것이 아니라 **첫 주 관찰 체계를 정교화하는 것**이다. "누가 어떤 유입 문장을 보고 들어와, 어떤 잡(J1~J4)에서 first value를 느꼈고, 7일 내에 second value(재가치)로 이어졌는가"를 사례 단위로 받아 적는 일이 먼저다. 표본이 한 자릿수~수십 명일 때는 퍼센트보다 사례 단위 분류가 더 정직하고, 그 기록이 출시 후 개선이 같은 방향으로 움직이는지를 보는 자기 기준선이 된다.

- Amplitude "B2B Product Benchmarks": https://info.amplitude.com/rs/138-CDN-550/images/b2b-product-benchmarks.pdf
- ProductLed "Product-Led Growth Benchmarks": https://productled.com/blog/product-led-growth-benchmarks

## §2 J1~J4 first value → second value(7일 내 재가치) 연결표

| 잡 | first value — 무엇을 느끼면 닫히나 (이벤트) | second value = 7일 내 재가치 (이벤트) |
|---|---|---|
| **J1 기록형** | "방금 한 일을 남겼다"는 즉시 만족 / 첫 `deed_saved` | "또 남길 수 있다"가 확인됨 / 다음 `deed_saved` |
| **J2 누적형** | "쌓였다"는 누적감 / 첫 `deed_saved` | 누적이 한 번 더 보임 / 다음 `deed_saved`, 보조로 `level_up_viewed` |
| **J3 AI 호기심형** | "AI가 내 행동을 이렇게 봤구나" / 첫 `deed_judged` (저장 전이라도 닫힘) | "다르게 보면?" 재확인 또는 저장으로 이어짐 / 다음 `deed_judged` 또는 `deed_saved` |
| **J4 회고형** | "나중에 돌아볼 게 생겼다"는 지연 가치 / 첫 `deed_saved` | "쌓이면 로그가 된다"가 확인됨 / 다음 `deed_saved` |

**보충 메모 — "7일 내 재가치" 컬럼의 의미.** 이 컬럼은 단순 재방문 수가 아니라 **second value(재가치)** 를 적는 자리다. 첫 주 리텐션의 전조는 "또 들어왔다"가 아니라 "또 들어와서 같은 잡의 가치를 한 번 더 느꼈다"이기 때문이다. 그래서 같은 `deed_saved`라도 J1·J2·J4는 가치 방향이 다르다(J1 즉시 만족 / J2 누적감 / J4 지연 가치) — 이벤트가 같다고 같은 재가치로 묶지 않는다. J3만 `deed_judged`에서 닫히고, 저장 없이 7일 안에 다시 들어와 한 번 더 판정을 본 것도 정상적인 second value다. **J3의 second value 부재나 judged−saved 갭을 이탈로 단정하지 않는다.** J2의 `level_up_viewed`는 누적 재가치의 보조 신호일 뿐, 첫 주에 레벨업에 닿지 않았다고 누적 가치가 없다고 보지 않는다. 위 매핑 값은 선행 문서(`first-session-jtbd-matrix.md`, `time-to-value-observation-brief.md`, `first-real-user-baseline-template.md`)에서 그대로 계승했으며 재정의하지 않는다.

## §3 첫 10~20명 관찰 행 템플릿

행은 **한 명당 한 행**이다. baseline 템플릿(`first-real-user-baseline-template.md` §2)의 컬럼을 계승하되, **7일 내 second value 도달 칸을 명시적으로 분리**해 첫 주 재가치를 따로 받아 적는다. 「예: …」 셀은 무엇을 적는지 보여주는 placeholder이며, 실제 작성 시 지우고 관찰값으로 채운다. 실명·연락처는 적지 않고 내부 식별자만 쓴다.

| # | 내부 식별자 | 유입 문장 (본인 표현 그대로) | 추정 잡 (J1/J2/J3/J4/혼합·미정) | first value (이벤트 + 시각) | 7일 내 second value (이벤트 + 시각, 없으면 미관찰) | 정성 관찰 메모 | synthetic/test 여부 | 판정 보류 메모 (해석 금지) |
|---|---|---|---|---|---|---|---|---|
| 1 | 예: U-001 | 예: "하루에 한 줄이라도 남겨봐"라는 지인 추천 | 예: J1 기록형 | 예: 첫 `deed_saved` / D1 14:03 | 예: 다음 `deed_saved` / D3 21:40 | 예: "사흘 뒤 다시 열어 '또 남겼다'며 안심하는 반응" | 예: 실사용 (제외 아님) | 예: 7일 내 second value 있었어도 '리텐션 확보'로 확정하지 말 것 |
| 2 | 예: U-002 | 예: "AI가 내 행동 점수 매겨준대"라는 SNS 캡처 | 예: J3 AI 호기심형 | 예: 첫 `deed_judged` / D1 21:11 | 예: 다음 `deed_judged` / D5 22:30 (저장은 여전히 없음) | 예: "닷새 뒤 다른 행동으로 다시 판정만 확인, 저장은 안 함" | 예: 실사용 (제외 아님) | 예: 저장 없는 재판정도 J3 정상 second value. judged−saved 갭을 이탈로 읽지 말 것 |

**컬럼 작성법 (한 줄 가이드)**

- **내부 식별자**: U-001 등 내부 식별자만. 실명·연락처·외부 공유 금지.
- **유입 문장**: 사용자 본인이 쓴 표현을 그대로 옮긴다(§4 참조). 모르면 "(미확인)", 짐작이면 끝에 "(추정)".
- **추정 잡**: 첫 세션의 주된 동기 하나. 헷갈리면 "혼합/미정"으로 둔다(억지 분류 금지).
- **first value**: 잡별 매핑은 §2를 따른다. 이벤트명 + 시각을 그대로 적고, 없으면 "미관찰".
- **7일 내 second value**: first value 이후 7일 안의 재가치 이벤트(잡별 매핑은 §2)와 시각을 적는다. 없으면 "미관찰". time gap 계산이 필요하면 `time-to-value-observation-brief.md` §2에 위임하고, 이 칸은 이벤트+시각만 받아 적는다.
- **synthetic/test 여부**: QA·내부 점검·더미 입력 등 실사용자가 아닌 트래픽은 **test (제외)**로 표시한다(판정 기준은 §7). 제외 행은 지우지 말고 남겨 둔다.
- **판정 보류 메모**: 성패/리텐션/PMF 결론을 쓰지 않는다. 해석하면 안 되는 점만 남긴다.

## §4 유입 문장 → 잡 → 반복 가치 연결 가이드

같은 잡이라도 어떤 **유입 문장(약속/말)** 을 보고 들어왔는지에 따라 first value에 닿는 경로가 다를 수 있다. "사소한 거 하나 남길 수 있다더라"를 듣고 온 J1과 "AI가 점수 매겨준대"를 보고 온 J1은, 같은 첫 `deed_saved`에 닿더라도 가는 길이 다르다. 유입 문장은 어떤 행동 이벤트보다 먼저 발생하는 **가장 앞단의 정성 신호**이며, 나중에 first value·7일 내 second value와 겹쳐 읽으면 "이 약속을 보고 온 사람이 첫 주에 어떤 재가치까지 갔나"를 한 줄로 잇게 해준다.

기록 방식은 단순하다. **사용자 본인이 쓴 표현을 그대로 옮긴다.** 우리 말로 다듬거나 마케팅 카피처럼 정돈하지 않는다(우리가 짓는 문장이 아니다). 직접 확인하지 못해 짐작으로 채운 칸은 반드시 끝에 "(추정)"을 붙여 관찰값과 추측을 구분하고, 전혀 모르면 "미확인"으로 비워 둔다 — 그럴듯한 문장을 지어 넣지 않는다.

채우는 출처도 못박는다. 이 칸은 **이미 허용된 대화나 내부 메모에서만** 받아 적는다. **이 칸을 채우는 일이 새로운 외부 접촉(메일·DM·설문·푸시·공개 모집)을 정당화하지 않는다.** 자연스럽게 이미 들어와 있는 말만 옮긴다. 그리고 첫 주 관찰에서 먼저 보는 것은 획득 채널의 숫자가 아니라 **"유입 문장과 잡의 정합성"** 이다 — 어떤 약속을 보고 온 사람이 그 약속과 같은 결의 가치를 첫 주에 느꼈는지(또는 다른 잡으로 갈렸는지)를 사례로 읽는다. 이 표는 사용자에게 내보낼 카피를 만드는 자리가 아니며(`copy-spec.md` 금지어 미사용), 화면·메시지 카피를 새로 정의하지 않는다.

## §5 prelaunch 해석 금지선

표본이 첫 10~20명일 때, 위 연결표와 관찰 행은 **관찰 렌즈일 뿐 합격/불합격 임계값이 아니다.** 선행 `activation-milestone-ladder.md` §4와 `pmf-response-analysis-rubric.md`의 작은 표본 금지선을 그대로 계승해, 아래는 *아직 판정하지 않는다*.

- **first value 도달률(setup→first value 전환)로 온보딩 성패를 선언하지 않는다.** `add_flow_started` 대비 `deed_saved`/`deed_judged` 비율이 낮아도 온보딩 실패로 결론내지 않는다.
- **7일 내 second value 도달률(first→second value)로 리텐션을 판정하지 않는다.** 재가치가 적게 보였다고 리텐션이 좋다/나쁘다를 선언하지 않는다.
- **judged−saved 갭을 곧장 이탈로 읽지 않는다.** J3는 저장 없이 `deed_judged`에서 가치를 얻고 정상 종료할 수 있다. 갭은 이탈이 아니라 잡 차이일 수 있다.
- **40% / 외부 임계값을 prelaunch에 적용하지 않는다.** 첫 10~20명 수준에서 비율로 PMF 달성/미달을 선언하지 않는다.
- **한 명의 강한 신호를 제품 방향으로 확정하지 않는다.** 표본이 작을수록 한 명의 인상적 경험·막힘이 과대평가되기 쉽다. 기록하되 방향 결정 근거로 굳히지 않는다.
- **작은 표본의 단계 도달을 세그먼트 크기로 읽지 않는다.** "second value 3명"을 J2 세그먼트 규모로 해석하지 않는다.

그래도 지금 할 수 있는 관찰: 첫 10~20명을 **사람·상황 단위**로 "어떤 유입 문장 → 어떤 잡 → first value → 7일 내 재가치"까지 갔는지 관찰표(§3)에 수기로 잇는다. 숫자 대신 "어떤 상황의 누가 어디서 멈췄나/한 번 더 돌아왔나"를 본다.

## §6 운영 경계: 503/가용성 이슈는 관찰 차단이지 마케팅 실패가 아니다

Virtue는 prelaunch이며, 라이브 페이지가 일시적으로 503을 반환하는 구간이 있다. 첫 주 활성화→리텐션 신호를 읽을 때 이 구간을 어떻게 다루는지가 이 표 전체 해석의 전제다.

**1. 가용성 문제는 "관찰 가능성 차단(observation blocker)"이지 마케팅 성패가 아니다.**
503·다운·심한 지연이 있는 동안에는 사용자가 들어와도 first value(`deed_saved`) / second value(7일 내 재방문 후 두 번째 `deed_saved` 또는 `level_up_viewed`) 이벤트가 **애초에 발생할 수 없다**. 그 앞단인 `add_flow_started`·`deed_judged`조차 발화되지 못한다. 따라서 이 구간에서 활성화·리텐션 신호가 낮게 나오더라도 이는 메시지·카피·유입 문장(잡 J1–J4 매칭)의 실패가 아니라 **제품이 관찰될 수 없었다는 사실**일 뿐이다. 이걸 마케팅 실패나 잡(Job) 실패로 해석하면 잘못된 개선 방향으로 끌려간다.

**2. 503이 지속되면 외부 유입·공개 액션을 키우지 않는다.**
가용성이 막힌 상태에서 유입을 늘리면 들어온 사람 대부분이 빈손으로 돌아가고, 관찰표는 "차단된 세션"으로만 채워져 **오염**된다. 이 마케팅 intent에서 인프라를 직접 고치지 않는다. 대신 라우터가 해당 건을 **제품/운영 쪽 "가용성 확인" Waiting 또는 별도 복구 작업으로 넘긴다**. 마케팅 측 행동은 "유입을 멈추고 가용성 복구를 기다린다" 한 가지다.

**3. 관찰표에서는 삭제하지 않고 표시한다.**
첫 10~20명 관찰 행 중 가용성 차단 구간에 걸린 세션은 정성 메모 칸에 **"가용성 차단(503 등)"**으로 표시하고, 비율·평균(예: first value→second value 도달률) 산정에서 **제외**한다. 행을 지우지 않는 이유는 두 가지다.
- 나중에 "왜 이 구간을 뺐는지" 추적할 수 있다.
- 잘못 제외했다고 판단되면 표시만 풀어 되돌릴 수 있다(되돌림 가능성 보존).

**4. 한 줄 운영 원칙**
> 가용성이 확인되기 전에는 첫 주 활성화/리텐션 신호를 판정하지 않는다.

## §7 관찰 위생: synthetic/test 트래픽 + 가용성 차단 구분

첫 10~20명은 표본이 작아서, 실제 사용자 한 명의 행이 비율 해석을 크게 흔든다. 그래서 집계 전에 **실제 사용자**와 (a) synthetic/test 트래픽, (b) 가용성 차단 구간을 사람 판단으로 먼저 갈라낸다. 새 이벤트·속성을 만들지 않고, 기존 4개 이벤트(`add_flow_started`, `deed_judged`, `deed_saved`, `level_up_viewed`)의 발화 패턴과 정성 정보만으로 구분한다.

| 구분 | 어떻게 알아채나 | 관찰표 처리 |
|---|---|---|
| 내부/QA 계정 | 운영자·개발자 본인이 아는 계정/디바이스, 알려진 테스트 시간대 | 삭제 아닌 "내부/QA" 표시 후 집계 제외, 사유 정성 메모 |
| 동일 디바이스 반복 `add_flow_started` | 짧은 시간 안에 같은 환경에서 `add_flow_started`만 여러 번 반복되고 `deed_saved`로 거의 이어지지 않음(반복 시도/점검 패턴) | 삭제 아닌 "반복 점검 추정" 표시 후 집계 제외, 사유 정성 메모 |
| 명백한 더미 입력 | 메모/입력에 "test", "asdf" 등 의미 없는 값, 또는 `deed_judged` 직전 명백한 placeholder 사진/텍스트 | 삭제 아닌 "더미 입력" 표시 후 집계 제외, 사유 정성 메모 |
| 봇/크롤러 | 사람 흐름과 무관하게 페이지만 스치고 `add_flow_started`조차 없음, 비정상적 접근 패턴 | 삭제 아닌 "봇/크롤러 추정" 표시 후 집계 제외, 사유 정성 메모 |
| 503/가용성 차단 구간 | 가용성 차단 시간대에 걸린 세션. 이벤트가 안 떴거나 도중에 끊김(§6 운영 경계 적용) | 삭제 아닌 "가용성 차단(503 등)" 표시 후 집계 제외, 사유 정성 메모 |

원칙은 다섯 행 모두 동일하다: **삭제하지 않고 표시한 뒤 집계에서만 제외하고, 사유를 정성 메모에 남긴다.** 신규 이벤트나 속성을 추가하지 않으며, 위 판단은 모두 기존 4개 이벤트의 발화 패턴 + 사람 판단으로만 한다. 자동 봇 필터링·계정 태깅 같은 측정 수단은 이 문서로 만들지 않는다.

## §8 post-launch 관찰 운영 리듬 (초안, 자동화는 Waiting)

런칭 이후 첫 10~20명을 따라갈 때의 **주 1회 가벼운 리듬**이다. 무겁게 굴리지 않고, 운영자가 손으로 관찰표를 갱신하는 수준으로 시작한다.

1. **실제 사용자 행 추가** — 그 주에 들어온 실제 사용자 세션을 관찰표에 행으로 추가한다.
2. **위생 플래그 먼저 채움** — 새로 추가한 행에 대해 synthetic/test 플래그와 가용성 차단(503 등) 플래그를 **먼저** 표시한다. (집계 전에 오염을 걸러내는 순서)
3. **first value → second value 갱신** — 각 실제 사용자 행에서 first value(첫 `deed_saved`, J3는 첫 `deed_judged`) 발생 여부와, **7일 내 second value** 도달(잡별 매핑은 §2) 여부를 갱신한다.
4. **지난주 "판단 보류" 재검토** — 지난주에 보류로 둔 행(애매한 synthetic/가용성 차단, 7일 창이 안 끝난 세션)을 다시 보고, 확정 가능하면 플래그를 갱신한다.

**자동화는 만들지 않는다.** 실제 대시보드 생성, 자동 집계, 정기 알림 발송은 모두 **Waiting / approval-needed**이며 본 문서로 만들지 않는다. 본 문서는 운영자가 수동으로 따라갈 수 있는 내부 관찰 리듬의 초안일 뿐이고, 코드·인프라·PostHog·대시보드·배포·권한·외부 발송 변경은 0건이다.
