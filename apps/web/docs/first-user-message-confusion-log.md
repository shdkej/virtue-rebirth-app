# Virtue 첫 사용자 메시지 혼란 로그

## 0. 목적 + 전제

### 목적

이 문서는 Virtue를 처음 본 사람이 제품을 **어떤 상자에 넣는지**와 **어디서 설명을 다시 요구하는지**를 한 줄씩 기록하기 위한 내부 로그 양식이다. 기존 기준선 문서들이 행동과 마찰을 잡았다면, 본 문서는 그 앞뒤에 붙은 사용자 언어를 보존한다. 목표는 공개 카피를 바로 고치는 것이 아니라, 첫 사용자 언어가 J1~J4 가치 해석과 어떻게 엇갈리는지 확인하는 것이다.

역할은 아래처럼 나눈다.

| 문서 | 역할 | 본 문서가 계승하는 것 |
|---|---|---|
| `first-real-user-baseline-template.md` | 첫 10~20명 1인 1행 기준선 | 실사용/test 분리, 작은 표본 금지선, J1~J4 first value 매핑 |
| `first-60-second-value-observation-script.md` | 첫 60초 행동 관찰 | 첫 가치에 닿기 전 사용자가 무엇을 먼저 이해했는지 보는 관찰 리듬 |
| `first-session-friction-observation-protocol.md` | 첫 세션 가시적 막힘 태그 | 되묻기/오해/저장 보류를 성패가 아니라 마찰 후보로 기록하는 태도 |
| `traffic-source-reading-boundary-table.md` | human/test/synthetic 판독 경계 | 사람 실사용, 메이커 self-test, synthetic/mock 트래픽을 섞지 않는 원칙 |

### 전제 못박기

1. **prelaunch 메시지 관찰표이지 성과 판정표가 아니다.** 이 표는 activation rate, conversion, retention, PMF, benchmark를 계산하지 않는다. 첫 10~20명 또는 그보다 작은 표본에서는 "몇 명이 이해했다"가 아니라 "어떤 말로 이해하거나 헷갈렸는가"만 남긴다.
2. **사용자 언어는 증거이지 결정 자체가 아니다.** 사용자가 붙인 제품명이나 되묻는 문장은 강한 신호지만, 한 사람의 표현으로 제품 포지션·공개 카피·세그먼트·기능 방향을 확정하지 않는다.
3. **새 트래킹 / 이벤트 / 속성 / 코드 / 카피 / 대시보드 / 세션리플레이 / 배포는 0이다.** 본 문서는 기존 관찰과 내부 기록을 정리하는 파일이다. PostHog 이벤트, session replay, dashboard, property, screen copy를 새로 만들지 않는다.
4. **외부 액션은 0이다.** 본 문서를 근거로 외부 패널, 유료 테스트, 설문, DM, 이메일, SNS, 공개 모집을 보내지 않는다. 이미 허용된 대화·관찰·내부 메모에서만 채운다.
5. **traffic-source 경계를 먼저 적용한다.** 사람 실사용, 메이커 self-test, synthetic/mock, 플랫폼 차이, 장래 agent/API 트래픽을 같은 메시지 증거로 섞지 않는다. synthetic/mock/self-test 발화는 표시 후 제외하거나 별도 메모로 둔다.

## 1. 왜 메시지 혼란을 따로 남기는가

Virtue는 아직 prelaunch라서, 숫자보다 먼저 "처음 본 사람이 무엇이라고 부르는가"가 중요하다. 약한 포지셔닝은 전환율보다 먼저 언어에서 드러난다. 예를 들면 사용자가 Virtue를 사진 일기, 습관 앱, 선행 기록장, AI 채점 놀이 중 어디로 분류하는지, 혹은 "그래서 이게 뭐예요?"라고 되묻는지가 그 신호다.

이 로그는 Wynter식 메시지 검증의 핵심 질문(이해되는가, 가치가 매력적인가, 의도한 인상이 전달되는가)과 April Dunford식 포지셔닝 질문(경쟁 대안·가치·고객·시장 상자가 맞는가)을 Virtue의 작은 표본에 맞게 줄인 것이다. 유료 패널이나 공개 카피 테스트를 하지 않고, 이미 들어온 사용자 표현만 내부적으로 구조화한다.

중요한 점은 **혼란을 실패로 읽지 않는 것**이다. "AI 채점 놀이인가요?"라는 말은 J3 AI 호기심형에는 좋은 입구일 수 있고, J1 기록형에는 기록 가치가 흐려졌다는 신호일 수 있다. 그래서 본 문서는 문장 하나를 곧장 좋다/나쁘다로 채점하지 않고, J1~J4 중 어떤 약속으로 해석됐는지 먼저 붙인다.

## 2. 메시지 혼란 로그 양식

행은 한 명 또는 한 관찰 단위다. 같은 사람이 다른 맥락에서 다시 설명을 요구하면 별도 행으로 남길 수 있지만, 표본 해석에서는 사람 수로 과대 계산하지 않는다.

| # | 관찰 대상 | traffic-source 판정 | 사용자가 붙인 제품명/대체재 | 되물은 문장 또는 혼란 표현 | 가장 먼저 이해한 가치 | J1~J4 해석 | 후속 카피 후보 여부 | 증거 출처 | 판정 보류 메모 |
|---|---|---|---|---|---|---|---|---|---|
| 1 | 예: U-001 | 예: 사람 실사용 | 예: 사진 일기 / 습관 앱 / AI 채점 놀이 / 선행 기록장 | 예: "그래서 사진을 올리면 뭐가 남는 거예요?" | 예: "AI가 내 행동을 봐준다" | 예: J3 AI 호기심형 | 예: 후보 있음 - J3 앞단 약속 설명 후보 | 예: 허용된 대화 메모 | 예: 한 명 표현으로 카피 변경 금지 |
| 2 | 예: U-002 | 예: 사람 실사용 | 예: 하루 기록 앱 | 예: "점수는 왜 필요한가요?" | 예: "오늘 한 일을 남긴다" | 예: J1 기록형 / J2 미정 | 예: 후보 보류 | 예: 60초 관찰 메모 | 예: 점수 이해 부족을 누적 가치 실패로 단정하지 말 것 |
| 3 | 예: QA-001 | 예: 메이커 self-test (제외) | 예: 테스트 입력 | 예: "mock 화면 확인" | 예: 없음 | 예: 미정 | 예: 없음 | 예: 내부 QA | 예: 사람 메시지 증거에 섞지 말 것 |
| 4 |  |  |  |  |  |  |  |  |  |
| ... |  |  |  |  |  |  |  |  |  |

### 컬럼 작성법

- **관찰 대상**: 내부 식별자만 쓴다. 실명, 연락처, 외부 공유 가능한 개인정보를 적지 않는다.
- **traffic-source 판정**: `traffic-source-reading-boundary-table.md`의 A~E 경계를 먼저 적용한다. 사람 실사용만 baseline 메시지 증거로 읽고, 메이커 self-test·synthetic/mock은 표시 후 제외한다.
- **사용자가 붙인 제품명/대체재**: 사용자가 실제로 붙인 말 그대로 쓴다. "사진일기", "습관앱", "AI 채점", "착한 일 기록장"처럼 제품을 넣은 상자다. 추정이면 반드시 "(추정)"을 붙인다.
- **되물은 문장 또는 혼란 표현**: 다시 설명을 요구한 문장, 헷갈린 표현, 다른 제품과 비교한 말을 그대로 적는다. 우리 말로 다듬어 카피처럼 바꾸지 않는다.
- **가장 먼저 이해한 가치**: 사용자가 먼저 잡은 약속이다. 기록, 누적, AI 판단, 회고 중 어느 가치가 먼저 보였는지 적되, 확실하지 않으면 "미정"으로 둔다.
- **J1~J4 해석**: 선행 문서의 잡 정의를 계승한다. 헷갈리면 "혼합/미정"으로 둔다.
- **후속 카피 후보 여부**: 공개 카피 변경이 아니라 내부 후보 여부만 적는다. "후보 있음", "보류", "없음" 중 하나로 충분하다. 후보가 있더라도 별도 승인 전에는 앱/랜딩/푸시/메일에 반영하지 않는다.
- **증거 출처**: 허용된 대화, 내부 관찰, 기준선 행, 60초 관찰 메모처럼 출처를 남긴다. 외부 발송으로 새로 수집하지 않는다.
- **판정 보류 메모**: 이 문장을 어떻게 과대해석하면 안 되는지 적는다.

## 3. J1~J4 해석 기준과 first value 매핑

본 문서는 first value 정의를 새로 만들지 않는다. `first-real-user-baseline-template.md`, `first-session-jtbd-matrix.md`, `activation-milestone-ladder.md`의 매핑을 그대로 따른다.

| 잡 | 사용자가 붙일 수 있는 상자 예시 | 메시지 혼란에서 볼 질문 | first value 매핑 |
|---|---|---|---|
| **J1 기록형** | 선행 기록장, 하루 기록 앱, 덕행 메모 | "뭘 얼마나 자세히 남겨야 해요?", "저장하면 뭐가 남아요?" | 첫 `deed_saved` |
| **J2 누적형** | 습관 앱, 점수 쌓기, 레벨업 게임 | "점수가 왜 필요해요?", "쌓이면 뭐가 달라져요?" | 첫 `deed_saved`, 보조 `level_up_viewed` |
| **J3 AI 호기심형** | AI 채점 놀이, AI가 보는 나, 평가 앱 | "AI가 뭘 기준으로 판단해요?", "저장 안 해도 결과만 보면 되나요?" | 첫 `deed_judged` |
| **J4 회고형** | 사진 일기, 나중에 보는 로그, 자기 회고장 | "나중에 어떻게 다시 봐요?", "기록이 모이면 뭐가 보여요?" | 첫 `deed_saved` |

> 고정 매핑: J1/J2/J4의 first value는 `deed_saved`이고, J3의 first value는 저장 전 `deed_judged`다. J3는 `deed_judged`에서 가치가 닫힐 수 있으므로 judged-without-saved를 곧장 이탈로 읽지 않는다.

## 4. 기존 관찰 문서와 연결하는 법

### `first-real-user-baseline-template.md`와 연결

baseline의 "유입 문장", "추정 잡", "정성 관찰 메모", "synthetic/test 여부"와 본 로그를 연결한다. 같은 사람의 행동 행과 메시지 행을 느슨하게 묶되, 본 문서가 baseline 원장을 대체하지 않는다. 행동은 baseline에, 메시지 혼란은 본 로그에 남긴다.

### `first-60-second-value-observation-script.md`와 연결

첫 60초 안에 사용자가 가장 먼저 말한 가치와 본 로그의 "가장 먼저 이해한 가치"를 맞춰 본다. 예를 들어 화면을 보자마자 "AI가 평가해주는 거네요"라고 이해했다면 J3 앞단 약속 신호로 남긴다. 반대로 결과 카드까지 간 뒤에야 의미를 알았다면 메시지 지연 후보로 둔다.

### `first-session-friction-observation-protocol.md`와 연결

되묻는 문장은 마찰 태그와 연결할 수 있다. "저장하면 뭐가 남아요?"는 저장 보류/결과 카드 이해 부족 후보일 수 있고, "AI가 어디 있어요?"는 J3 앞단 약속 공백 후보일 수 있다. 단, 태그 연결은 관찰 보조일 뿐, 한 문장으로 마찰 원인을 확정하지 않는다.

### `traffic-source-reading-boundary-table.md`와 연결

메시지 증거도 traffic-source 분류를 먼저 한다. 사람 실사용이 아닌 QA, 데모, mock, 내부 self-test, synthetic traffic, 장래 agent/API 호출은 사람 메시지 baseline에 섞지 않는다. 특히 synthetic/mock/self-test에서 나온 "이해했다/헷갈렸다"는 제품 포지셔닝 증거가 아니라 테스트 맥락 메모다.

## 5. 해석 금지선

아래는 본 문서로 하지 않는다.

- **activation rate / conversion / retention / PMF / benchmark 산출 금지.** 작은 표본의 메시지 혼란을 비율이나 합격선으로 바꾸지 않는다.
- **한 명의 언어로 공개 카피 변경 금지.** 후보는 남길 수 있지만 앱 카피, 랜딩 카피, 푸시, 메일, SNS 문구에 반영하지 않는다.
- **외부 패널·설문·DM·유료 테스트 금지.** Wynter식 검증은 참고 렌즈일 뿐, 본 Intent로 외부 발송이나 비용 집행을 하지 않는다.
- **신규 analytics 금지.** 새 이벤트, 속성, session replay, dashboard, cohort, PostHog view, tracking plan을 만들지 않는다.
- **traffic-source 혼합 금지.** 사람 실사용 언어와 synthetic/mock/self-test 언어를 같은 근거로 섞지 않는다.
- **J1~J4 first value 재정의 금지.** J1/J2/J4=`deed_saved`, J3=`deed_judged` 매핑을 유지한다.
- **사용자 언어를 사실 확정으로 오독 금지.** 사용자가 "습관 앱"이라고 불렀다고 Virtue가 습관 앱으로 확정되는 것이 아니다. 그것은 사용자가 처음 넣은 상자라는 증거다.

## 6. 후속 카피 후보를 남기는 방식

후속 카피 후보는 내부 메모까지만 허용한다. 아래처럼 "어떤 혼란을 줄이기 위한 후보인지"만 적고, 문구 자체를 공개 표면에 반영하지 않는다.

| 혼란 패턴 | 가능한 내부 후보 메모 | 실행 상태 |
|---|---|---|
| J3 AI 약속이 `/add` 전 보이지 않음 | "AI가 행동을 어떻게 보는지"를 첫 설명 후보로 보관 | proposal-only |
| J2 누적 가치가 점수 놀이라고만 이해됨 | 누적이 레벨업이 아니라 "쌓이는 자기 기록"이라는 설명 후보 보관 | proposal-only |
| J4가 사진 일기와만 섞임 | 사진보다 행동 회고가 중심이라는 설명 후보 보관 | proposal-only |
| J1 기록 가치가 너무 약하게 이해됨 | "오늘 한 일 하나를 남긴다"는 최소 행동 설명 후보 보관 | proposal-only |

이 표는 카피 저장소가 아니라 관찰에서 나온 후보의 위치만 남기는 곳이다. 실제 copy-spec 변경, 화면 반영, 배포, 메시지 발송은 별도 Intent와 승인이 필요하다.
